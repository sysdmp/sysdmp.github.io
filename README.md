# ddda-build-solver (web)

A serverless, browser-only MILP solver for Dragon's Dogma™: Dark Arisen level-200
builds. This is the **primary** implementation going forward; the Python prototype
in `pycli/` is a finished reference.

The solver core is a **library** (`src/solver.js`): a function you call with
options, returning a build object. `src/app.js` + `src/index.html` are the UI; the
build inlines everything into a single self-contained `index.html` at the repo root.

## Features

The objective maximizes a **balanced** weighted stat total (hp/st discounted to
0.1 vs 1.0 for the combat stats), and supports these specifiers:

- **Per-stat min/max bounds** — `min === max` requests an exact value.
- **Divisor rounding** — force a stat to a multiple of N (max dropped, min kept).
- **Match** — tie two stats: `=` equal, `~` within a tolerance (10 for combat
  pairs, 100 for hp↔st). hp/st pair only with each other; combat stats with any
  other combat stat.
- **Bias** — soft per-stat preference (−5..+5); bends the build, never overrides
  a bound.
- **Pawn mode** — excludes the hybrid (Arisen-only) vocations and forces ≥1 of
  the 1→10 levels into the starting vocation.
- **Weight class** (SS–LL) — sets level-1 stamina.

Files:

Source lives in `src/`; the deployable site is built from it:

- `src/data.js` — growth tables + helpers (`statsOf`, `growth`, weight/match data,
  per-stat max reachable values). Patched (non-vanilla) Magick Archer values.
- `src/solver.js` — `solveMaxTotal(highs, opts)`: one MILP per basic start vocation
  (integer level-counts per (vocation, tier), block-size constraints), maximizes
  the balanced objective subject to the specifiers, returns the best build.
- `src/highs-loader.js` — initializes HiGHS with its wasm inlined as bytes.
- `src/app.js` — the browser UI controller (vocation picker, stat targets, weight
  selector, shareable URL).
- `src/index.html` — the page template (CSS inline, a placeholder for the bundle).
- `src/build.mjs` — the build: bundles `src/` and inlines it into the template.
- `src/test.mjs` — the Node test harness.
- `index.html` — the **built single-file app** (generated by `make`; see Building).

## Usage (library)

```js
import { loadHighs } from './src/highs-loader.js';
import { solveMaxTotal } from './src/solver.js';

const highs = await loadHighs();
const build = solveMaxTotal(highs, {
  bounds: { hp: { min: 3500 }, attack: { divisor: 100 } },
  match: [{ a: 'attack', b: 'mattack', tol: 0 }],
  bias: { mattack: 3 },
  weight: 'LL',
  pawn: false,
});
// -> { start, counts: { to10, to100, to200 }, stats, total }
```

`solveMaxTotal` is synchronous once `highs` is initialized. See its JSDoc for the
full option list.

## Solver backend

[HiGHS](https://github.com/lovasoa/highs-js) — a WebAssembly MILP solver (MIT).
Chosen for its CBC-class cut generation: it solves coarse-divisor cases (e.g.
`st÷3333`, where only one multiple is reachable) in milliseconds, whereas simpler
branch-and-bound solvers stall on them. Problems are passed as CPLEX LP-format
strings, which `src/solver.js` builds.

## Building

`make` bundles `src/` — the solver, data, and the HiGHS factory with its `.wasm`
**inlined as base64** — with [esbuild](https://esbuild.github.io/), then splices
that minified JS into the `src/index.html` template (whose CSS is already inline).
The output is **one self-contained `index.html`** at the repo root: no separate
`.js`/`.wasm`, no `node_modules`, no runtime fetches. Drop that single file on
gh-pages.

```sh
npm install
make            # -> index.html (~4 MB, everything inlined)
make clean      # remove the built index.html
```

## Running the page

Because everything is inlined into one file, just open it — no server needed:

```sh
make
open index.html         # or double-click it; works over file://
```

A static server (`python3 -m http.server`) also works if you prefer, but isn't
required.

## Testing

The objective is linear and separable per tier, so the optimum has a closed form
(each tier takes all its levels in the highest weighted-gain vocation). The tests
use that as an independent oracle, plus structural invariants and one check per
specifier (bounds, divisor incl. st÷3333, match, bias, pawn, weight):

```sh
npm install
npm test        # node src/test.mjs
```

### Cross-validation against the Python prototype

`npm run test:py` treats the Python prototype (`pycli/ddda-build-solver.py`, run
via `uv`) as the source of truth and asserts the web solver reaches the **same
optimal objective** on matched inputs — the balanced weighted score, the exact
maximized stat for `--maximize`, or the distinct-vocation count for
minimize-vocations. (Allocations may differ when builds tie at the optimum; only
the objective value is compared. The web solver runs HiGHS with a zero MIP gap so
both prove exact optima.) It has a **static** suite (fixed cases) and a
**dynamic** fuzz suite (random inputs each run):

```sh
npm run test:py
DDDA_SEED=42 DDDA_FUZZ=30 npm run test:py   # replay a seed / set fuzz count
```

Only **bias** is excluded from cross-validation — it diverges from the prototype
on purpose (web uses a soft weight nudge, Python an equal-share floor). `--nice`
is not ported. Everything else, including hp↔st `~` matches (tol 100 on both
sides) and no-early-switcheroo, is cross-checked. See `src/bias-study.mjs` for a
characterization of the bias difference.
